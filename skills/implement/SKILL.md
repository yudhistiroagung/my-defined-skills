# Task Implementer

Implements one task at a time from a `tasks/` directory. Does not advance to the next
task until the current one is explicitly confirmed done by the user.

---

## Core Rules

1. **Scan existing codebase.** use skill `codebase-scan` before starting any task.
2. **One task at a time.** Never start a second task while one is `in_progress`.
3. **Dependency guardrail.** A task can only be picked if ALL its `depends_on` tasks
   are `✅ done` in `_index.md`. If any dependency is not done, refuse and name the blockers.
4. **User confirms done.** The agent never self-promotes a task to `done`.
   Only mark `done` when the user explicitly says so.
5. **Status lives in `_index.md` only.** Never edit the status inside a task file.

---

## Step 1 — Read the Task Index

Before doing anything, read `tasks/_index.md` to get the current state:
- Which tasks are `✅ done`
- Which tasks are `🔵 in_progress`
- Which tasks are `⬜ todo`
- The `depends_on` list for each task

> If `tasks/_index.md` does not exist, stop and tell the user to run `split-prd` first.

---

## Step 2 — Pick a Task

### If the user named a specific task (e.g. "work on task-003"):

Run the dependency check:

```
FOR each task_id in task-003.depends_on:
  IF _index status of task_id != "done":
    BLOCK — do not proceed
```

If blocked, respond:

> ❌ **task-003 is blocked.**
> The following tasks must be completed first:
> - `task-001` — ⬜ todo *(reason from Dependencies Detail)*
> - `task-002` — 🔵 in_progress *(reason from Dependencies Detail)*
>
> Finish those tasks first, then come back to task-003.

If clear, proceed to Step 3.

### If the user said "pick the next task" or similar:

Find all `⬜ todo` tasks where every `depends_on` entry is `✅ done`. These are the **available tasks**.

If multiple tasks are available, present them and ask the user to choose:

> ✅ **Available tasks (no blockers):**
> | Task | Title | Estimate | Wave |
> |------|-------|----------|------|
> | task-002 | Build CRUD API | M | 2 |
> | task-003 | Build UI form | S | 2 |
>
> Which one would you like to work on?

If only one task is available, confirm before starting:

> The next available task is **task-002: Build CRUD API** (M, Wave 2).
> Shall I start on this?

If NO tasks are available but some are still `⬜ todo`:

> ⏳ All remaining tasks are blocked by unfinished dependencies.
> Complete the following in-progress tasks first:
> - `task-001` — 🔵 in_progress

If ALL tasks are `✅ done`:

> 🎉 All tasks are complete!

---

## Step 3 — Load and Announce the Task

Read the full `tasks/task-XXX.md` file. Update `_index.md` status to `🔵 in_progress`.

Announce to the user:

> ## 🔵 Starting: task-XXX — {title}
> **Goal:** {goal from task file}
> **Estimate:** {estimate}
> **Acceptance Criteria:**
> - [ ] criterion 1
> - [ ] criterion 2
>
> Starting implementation now…

---

## Step 4 — Implement the Task

Work through the task file fully:

- Follow the **Acceptance Criteria** as the definition of done — implement everything listed
- Read **Technical Notes** for implementation hints
- Respect **Out of Scope** — do not implement anything listed there, even if it seems useful
- If an **Open Question** blocks implementation, stop and ask the user before continuing

Use the project's existing architecture skills as needed during implementation:
- New domain layer → `create-domain-layer`
- New data layer → `data-layer-scaffold`
- New local datasource → `create-local-datasource`
- New remote datasource → `create-remote-datasource`
- Repository implementation → `create-repository-implementation`

---

## Step 5 — Report and Wait for Confirmation

When implementation is complete, give a summary and **stop**. Do not pick the next task.

> ## ✅ Implementation complete: task-XXX — {title}
>
> **What was done:**
> - {brief summary of each acceptance criterion delivered}
>
> **Files changed:**
> - `src/...` — {what changed}
>
> **Acceptance Criteria:**
> - [x] criterion 1
> - [x] criterion 2
>
> Please review and test. When you're happy, say **"mark task-XXX as done"** and I'll
> move on to the next available task.

---

## Step 6 — Handle User Confirmation

### User confirms done ("mark task-XXX as done", "it's good", "looks good"):

1. Update `_index.md` — set task-XXX to `✅ done`
2. Compute which tasks just became unblocked (tasks whose `depends_on` are now all `✅ done`)
3. Announce:

> ✅ **task-XXX marked as done.**
>
> **Newly unblocked tasks:**
> | Task | Title | Estimate |
> |------|-------|----------|
> | task-004 | Wire UI to API | M |
>
> Say "pick the next task" to continue, or name a specific task to work on.

### User requests rework ("this needs changes", "reopen task-XXX"):

Keep status as `🔵 in_progress` (or set back to it) and ask what needs fixing:

> What needs to change? I'll update the implementation accordingly.

### User asks for status:

Print the Task Summary table from `_index.md` with current badges.

---

## Guardrail Reference

The dependency check runs on every pick attempt. It is not skippable.

```
BEFORE starting any task:
  READ _index.md
  FOR each dep in task.depends_on:
    IF dep.status != "done" → BLOCK with explanation
  IF no blockers → proceed
```

| Situation | Response |
|-----------|----------|
| Task named, all deps done | Proceed to implementation |
| Task named, any dep not done | Block, list unfinished deps with their current status |
| "Next task", multiple available | List options, ask user to choose |
| "Next task", one available | Confirm, then proceed |
| "Next task", all todo tasks blocked | List what needs to finish first |
| All tasks done | Celebrate, stop |

---

## Status Badge Reference

| Status | Badge | Set by |
|--------|-------|--------|
| `todo` | `⬜ todo` | split-prd (initial) |
| `in_progress` | `🔵 in_progress` | Agent (on pick) |
| `done` | `✅ done` | **User only** (never agent) |
